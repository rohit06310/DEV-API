--------------------------------------------------------------------------------
  1. PROJECT SUMMARY
--------------------------------------------------------------------------------

DevFlow is a Streamlit-based web application that helps developers, students,
and researchers understand and integrate REST APIs faster. Given any API
documentation URL, DevFlow automatically:

  • Scrapes and parses the documentation page
  • Uses an AI model (Groq / LLaMA 3.3) to extract structured API metadata
  • Presents endpoints, authentication details, and use cases in a clean UI
  • Generates production-ready SDK code in 6 programming languages
  • Provides a live HTTP sandbox to test API endpoints in real time

Application Name  : DevFlow – API Integration Accelerator
Framework         : Streamlit >= 1.35.0
AI Backend        : Groq (LLaMA-3.3-70b-versatile, fallback: llama3-8b-8192)
Entry Point       : app.py
Live URL (local)  : http://localhost:8501

── app.py (31 KB, 645 lines) ──────────────────────────────────────────────────
  The single entry point for Streamlit. Intentionally thin — all business logic
  delegates to utils/. Responsibilities:
    • st.set_page_config() — page title, icon, layout
    • Session state initialisation (_DEFAULTS dict)
    • @st.cache_data wrappers (_cached_parse, _cached_sdk) with 10-min TTL
    • UI helpers: HTTP method badge, URL-normaliser set_url_and_analyze()
    • Global CSS injection (Inter font, tab styles, endpoint rows, log rows)
    • Sidebar: Groq status, bookmarks, recent history, how-to guide
    • Header: DevFlow title + tagline
    • Input area: URL text input, use case textarea, Analyze / Clear buttons
    • Main router:
        - is_analyzing → calls display_results()
        - has_analyzed → re-renders cached results
        - else → landing page (Quick Start + feature cards)
    • display_results() — renders all 6 tabs from a parsed APISpec

── utils/__init__.py (825 B) ───────────────────────────────────────────────────
  Public API surface for the utils package. All imports in app.py and components
  come from here. Exports:
    chat_completion, get_client, GROQ_MODEL_PRIMARY,
    fetch_doc_text, safe_domain,
    parse_api_docs, APISpec,
    generate_sdk, SUPPORTED_LANGUAGES, LANG_EXTENSIONS

── utils/groq_client.py (2.9 KB, 81 lines) ────────────────────────────────────
  Singleton Groq client configuration. ALL LLM calls in the project must use
  chat_completion() from this module.
    • GROQ_MODEL_PRIMARY  = "llama-3.3-70b-versatile"  (128k context window)
    • GROQ_MODEL_FALLBACK = "llama3-8b-8192"            (smaller, faster)
    • get_client() → returns Groq client or None if key missing
    • chat_completion(messages, json_mode, max_tokens, temperature, use_fallback)
      → Wraps Groq completions API; auto-retries with fallback model on failure

── utils/scraper.py (2.7 KB, 84 lines) ────────────────────────────────────────
  Documentation fetcher and HTML-to-text converter.
    • fetch_doc_text(url, max_chars=14000, timeout=8)
        1. HTTP GET with browser-like User-Agent to bypass bot blocks
        2. BeautifulSoup (lxml backend) parses HTML
        3. Strips all noise tags: script, style, nav, footer, iframe, etc.
        4. Prefers <main>, role="main", <article>, or content-class divs
        5. Returns whitespace-collapsed plain text (max 14,000 chars)
    • safe_domain(url) → extracts netloc from a URL safely

── utils/parser.py (7.6 KB, 200 lines) ────────────────────────────────────────
  Core AI-powered documentation parser. Single source of truth for all
  structured API data in the application.
    TypedDicts:
      Endpoint   — { method, path, description, params }
      APISpec    — { api_name, version, base_url, description, auth_method,
                     auth_headers, auth_example, rate_limits, endpoints,
                     use_cases, research_notes }
    Functions:
      _normalise_endpoint(raw)   → ensures all 4 Endpoint fields present
      _validate_spec(data)       → fills missing fields with safe defaults
      _strip_json_fences(text)   → removes accidental ```json fences
      parse_api_docs(url, use_case) → full pipeline:
          1. fetch_doc_text() scrapes the page
          2. Builds LLM prompt (JSON schema + strict rules)
          3. Calls chat_completion(json_mode=True, max_tokens=4096, temp=0.1)
          4. Parses JSON, validates, returns APISpec

── utils/sdk_generator.py (5.4 KB, 147 lines) ──────────────────────────────────
  AI-powered SDK and wrapper code generator. Completely decoupled from UI.
    SUPPORTED_LANGUAGES: Python, JavaScript, TypeScript, Java, Go, cURL
    LANG_EXTENSIONS:     py, js, ts, java, go, sh
    Styles:
      "Minimal Functions" → standalone functions, no classes, minimal setup
      "Wrapper Class"     → OOP client class with constructor, typed methods,
                            docstrings, HTTP error handling
    Functions:
      _build_prompt(...)         → constructs the LLM prompt for code gen
      _clean_code_output(text)   → strips any accidental markdown fences
      generate_sdk(api_name, base_url, auth_headers, lang, style,
                   endpoint_target)
          → calls chat_completion(json_mode=False, max_tokens=2048, temp=0.15)
          → returns raw source code string

── utils/sdk_templates.py (27 KB) ──────────────────────────────────────────────
  Static reference code templates. Pre-written boilerplate and helper snippets
  used alongside or instead of AI generation for common patterns.

── utils/session.py (7.1 KB, 112 lines) ───────────────────────────────────────
  Session state management and preset API catalog.
    PRESET_APIS: hardcoded reference data for 3 popular APIs:
      • GitHub REST API  — 247 endpoints, Bearer Token / OAuth 2.0
      • Stripe API       — 312 endpoints, HTTP Basic Auth (API Key)
      • Twilio API       —  89 endpoints, HTTP Basic Auth (Account SID + Token)
    init_session_state() → initialises all st.session_state keys with defaults


--------------------------------------------------------------------------------
  4. APPLICATION TABS (UI)
--------------------------------------------------------------------------------

After analyzing an API URL, DevFlow renders 6 tabs:

  Tab 1 — 📊 Overview
    • API name, description (info box)
    • 4 metric cards: Version, Endpoint count, Auth method, Rate limit
    • Common use cases displayed as pills
    • Base URL code block

  Tab 2 — 🔗 Endpoints
    • Live search filter (path, description, method)
    • Method dropdown filter (All / GET / POST / PUT / DELETE / PATCH)
    • Colour-coded HTTP method badges per row
    • Endpoint path + description + params per row
    • Export all endpoints as CSV download

  Tab 3 — 🔐 Authentication
    • Auth type label
    • Required headers in JSON code block
    • cURL example command
    • Python (requests) and JavaScript (fetch) ready-to-run snippets

  Tab 4 — ⚙️ SDK (Code Generator)
    • Select target endpoint (or "Full API Integration")
    • Select language: Python / JavaScript / TypeScript / Java / Go / cURL
    • Select style: Minimal Functions or Wrapper Class
    • "Generate Code" button → calls Groq LLM → displays code
    • Download generated code as a language-appropriate file

  Tab 5 — 📖 Examples
    • Python quick-start snippet (generic call() wrapper)
    • JavaScript fetch snippet
    • Per-endpoint Python snippets for first 8 endpoints (expandable)

  Tab 6 — 🧪 Sandbox (Live HTTP Tester)
    • Form: Method selector + URL field
    • Headers (JSON), Query Params (JSON), Body (JSON) text areas
    • "Send Request" → fires real HTTP request (timeout: 15s)
    • Response: status code, reason, latency (ms), size (bytes)
    • Response headers (expandable) + response body (JSON or raw text)
    • Session request log (last 10 requests, clearable)

  8. HOW TO RUN
--------------------------------------------------------------------------------

  # 1. Clone / navigate to project directory
  cd "Dev API"

  # 2. Create and activate virtual environment
  python -m venv venv
  source venv/bin/activate          # Linux / macOS
  venv\Scripts\activate             # Windows

  # 3. Install dependencies
  pip install -r requirements.txt

  # 4. Set environment variable
  echo "GROQ_API_KEY=your_key_here" > .env

  # 5. Run the app
  streamlit run app.py

  # App will be available at: http://localhost:8501

  import requests
  from typing import Optional, Dict, Any


  class JSONPlaceholderClient:
      """Client for the JSONPlaceholder REST API."""

      BASE_URL = "https://jsonplaceholder.typicode.com"

      def __init__(self):
          self.session = requests.Session()
          self.session.headers.update({"Content-Type": "application/json"})

      def _request(self, method: str, path: str, **kwargs) -> Any:
          url = f"{self.BASE_URL}{path}"
          response = self.session.request(method, url, **kwargs)
          response.raise_for_status()
          return response.json()

      def get_posts(self, user_id: Optional[int] = None) -> list:
          """Retrieve all posts, optionally filtered by userId."""
          params = {}
          if user_id is not None:
              params["userId"] = user_id
          return self._request("GET", "/posts", params=params)

      def get_post(self, post_id: int) -> Dict:
          """Retrieve a specific post by ID."""
          return self._request("GET", f"/posts/{post_id}")

      def create_post(self, title: str, body: str, user_id: int) -> Dict:
          """Create a new post. Returns the created object with a fake ID."""
          payload = {"title": title, "body": body, "userId": user_id}
          return self._request("POST", "/posts", json=payload)

      def update_post(self, post_id: int, title: str, body: str) -> Dict:
          """Partially update a post by ID."""
          payload = {"title": title, "body": body}
          return self._request("PATCH", f"/posts/{post_id}", json=payload)

      def delete_post(self, post_id: int) -> Dict:
          """Delete a post by ID. Returns an empty dict on success."""
          return self._request("DELETE", f"/posts/{post_id}")


  # Usage
  client = JSONPlaceholderClient()
  posts = client.get_posts(user_id=1)
  print(posts[0])

================================================================================
  14. DETAILED PROJECT EXPLANATION
================================================================================

DevFlow is an AI-powered API Integration Accelerator built with Python and
Streamlit. Its core mission is to eliminate the friction developers face when
first encountering an unfamiliar REST API. Instead of reading through pages of
documentation manually, DevFlow lets you paste a single URL and instantly
receive a structured, interactive breakdown of the API — complete with working
code you can run immediately.

This section provides a deep, comprehensive explanation of every aspect of the
project: its design philosophy, feature mechanics, technical decisions, and the
reasoning behind each architectural choice.


────────────────────────────────────────────────────────────────────────────────
  14.1  WHY DEVFLOW EXISTS — THE PROBLEM IT SOLVES
────────────────────────────────────────────────────────────────────────────────

Every developer who integrates a new API faces the same repetitive workflow:

  1. Find the documentation URL
  2. Hunt for authentication details (what headers? which token type?)
  3. Identify which endpoints are relevant to the task
  4. Manually write boilerplate HTTP client code
  5. Test requests one at a time in a terminal or Postman
  6. Debug until it works, then clean up the code

This process can take anywhere from 30 minutes to several hours per API.
DevFlow collapses that entire workflow into under 60 seconds by automating
every step with AI. The result is not a summary — it is actionable, runnable
output that a developer can drop directly into their codebase.

  Target users:
    • Backend/Frontend developers integrating third-party APIs
    • Students learning how REST APIs work
    • Researchers who need quick access to public data APIs
    • Hackathon participants who need to move fast
    • Technical writers documenting integrations


────────────────────────────────────────────────────────────────────────────────
  14.2  HIGH-LEVEL ARCHITECTURE — HOW THE PIECES FIT
────────────────────────────────────────────────────────────────────────────────

DevFlow follows a strict separation-of-concerns architecture:

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                          PRESENTATION LAYER                             │
  │  app.py  ←→  components/  ←→  config/theme.py                          │
  │  Streamlit UI, routing, caching wrappers, CSS injection                 │
  └────────────────────────────┬────────────────────────────────────────────┘
                               │ calls
  ┌────────────────────────────▼────────────────────────────────────────────┐
  │                           BUSINESS LOGIC LAYER                          │
  │  utils/scraper.py   — fetches and cleans HTML doc text                  │
  │  utils/parser.py    — converts raw text → structured APISpec via LLM    │
  │  utils/sdk_gen.py   — converts APISpec → source code via LLM            │
  │  utils/session.py   — manages Streamlit session state & preset catalog  │
  │  utils/groq_client.py — singleton LLM client with auto-fallback         │
  └────────────────────────────┬────────────────────────────────────────────┘
                               │ calls
  ┌────────────────────────────▼────────────────────────────────────────────┐
  │                           EXTERNAL SERVICES                              │
  │  Groq API (LLaMA 3.3 70B) — LLM inference for parsing + code gen        │
  │  Target API servers        — Live HTTP calls made from the Sandbox tab   │
  └─────────────────────────────────────────────────────────────────────────┘

Key principle: app.py is intentionally thin. It does not contain business
logic. Every real computation happens in utils/. This makes the codebase
testable, modular, and easy to maintain. The UI layer can be swapped entirely
(e.g., replaced with a FastAPI + React frontend) without touching utils/.


────────────────────────────────────────────────────────────────────────────────
  14.3  THE AI PIPELINE — STEP BY STEP
────────────────────────────────────────────────────────────────────────────────

DevFlow's intelligence is built on a two-stage LLM pipeline:

  STAGE 1 — DOCUMENTATION PARSING  (utils/parser.py)
  ────────────────────────────────────────────────────

  Input:  Any API documentation URL
  Output: A fully validated APISpec dictionary (Python TypedDict)

  Step 1 — Fetch (utils/scraper.py → fetch_doc_text)
    • Sends an HTTP GET request with a browser-like User-Agent header to avoid
      being blocked by anti-bot protections on documentation sites.
    • Uses BeautifulSoup with the lxml backend (fastest available) to parse HTML.
    • Aggressively strips all noise: <script>, <style>, <nav>, <footer>,
      <header>, <iframe>, <aside>, ads, and cookie banners.
    • Intelligently focuses on content: prefers <main>, role="main", <article>,
      and divs with class names containing "content", "docs", "body", or "main".
    • Collapses all whitespace and returns the first 14,000 characters of
      clean plain text — enough context for the LLM without exceeding limits.

  Step 2 — LLM Extraction (utils/parser.py → parse_api_docs)
    • Builds a structured prompt containing:
        - The cleaned documentation text
        - A strict JSON output schema (the APISpec TypedDict)
        - Rules preventing hallucination (e.g., "return empty string if unknown")
        - The optional user-provided use case for targeted extraction
    • Calls chat_completion() with json_mode=True, max_tokens=4096, temp=0.1
      (low temperature = deterministic, factual extraction, minimal creativity)
    • LLM: LLaMA 3.3 70B (128K context window) via Groq's inference API

  Step 3 — Validation (_validate_spec)
    • Checks every field in the returned JSON for presence and correct type
    • Fills all missing fields with safe, descriptive defaults rather than
      crashing — this ensures the app always shows something useful
    • Normalises each endpoint object to guarantee all 4 required fields exist:
      method, path, description, params

  APISpec structure (what gets extracted):
    {
      "api_name":      string   — Human-readable API name
      "version":       string   — API version (e.g. "v3", "2024-06-20")
      "base_url":      string   — Root URL (e.g. "https://api.github.com")
      "description":   string   — 1-3 sentence description of the API
      "auth_method":   string   — Auth type (e.g. "Bearer Token", "API Key")
      "auth_headers":  object   — Exact headers required for authentication
      "auth_example":  string   — A ready-to-run cURL command with auth
      "rate_limits":   string   — Rate limiting rules as described in docs
      "endpoints":     array    — List of Endpoint objects (see below)
      "use_cases":     array    — Common use case strings (shown as pill tags)
      "research_notes":string   — Additional notes relevant to the use case
    }

    Each Endpoint:
    {
      "method":      string  — HTTP verb (GET, POST, PUT, DELETE, PATCH)
      "path":        string  — URL path template (e.g. "/repos/{owner}/{repo}")
      "description": string  — What the endpoint does
      "params":      string  — Comma-separated list of query/path parameters
    }

  STAGE 2 — SDK CODE GENERATION  (utils/sdk_generator.py)
  ─────────────────────────────────────────────────────────

  Input:  API name, base URL, auth headers, target language, style, endpoint
  Output: Raw source code string (no markdown, no fences, no explanations)

  The SDK generator is invoked only when the user clicks "Generate Code" in
  the SDK tab. It is a separate LLM call with completely different prompting:

  • System prompt: Defines the LLM as a "senior software engineer" who writes
    production-quality code and NEVER outputs markdown, preamble, or commentary.
  • User prompt: Contains the API context + strict output rules:
      1. Output ONLY raw <language> code
      2. NO markdown code fences (no backticks anywhere)
      3. NO introductory sentences, NO closing remarks
      4. Handle HTTP errors with clear error messages
      5. Include import statements at the top
  • Temperature: 0.15 (very low — favours correct, conventional code patterns)
  • max_tokens: 2048 (sufficient for a full SDK class in any language)

  Two generation styles:

  "Minimal Functions"
    → Standalone functions, no classes, minimal setup
    → Best for scripts, quick integrations, and beginners
    → Output: a set of directly callable functions

  "Wrapper Class"
    → Object-oriented client class with constructor, typed methods,
      docstrings, and HTTP error handling
    → Best for production codebases and library-style integrations
    → Output: a full class that can be imported and instantiated

  Post-processing (_clean_code_output):
    • Strips any accidental ```language fences that the LLM might add
    • Removes trailing backticks
    • Strips all remaining backtick characters

  14.4  FEATURE DEEP-DIVE — EVERY CAPABILITY EXPLAINED
────────────────────────────────────────────────────────────────────────────────

  ── FEATURE 1: AI-Powered Documentation Parsing ────────────────────────────

  What it is:
    The engine of DevFlow. Given any URL pointing to a REST API's documentation,
    the parser uses an LLM to extract all structured metadata in seconds.

  Why it matters:
    API documentation formats vary wildly across providers. Some use OpenAPI
    specs, some use Markdown, some use custom HTML renderers. A rule-based
    parser would need to handle dozens of formats. The LLM approach is format-
    agnostic — it reads natural language the same way a human does.

  Technical detail:
    • 14,000 character context window fed to the LLM at once
    • JSON mode enabled (forces the LLM to return valid JSON, never plain text)
    • Schema-enforced output prevents hallucination of unknown fields
    • Low temperature (0.1) ensures factual extraction, not creative invention

  What it extracts:
    • API name, version, base URL, description
    • Full endpoint list (method, path, description, params)
    • Authentication method and required headers
    • A ready-to-run cURL authentication example
    • Rate limiting rules
    • Common use cases (shown as colour-coded pills in the Overview tab)

────────────────────────────────────────────────────────────────────────────────
  14.5  SESSION STATE MANAGEMENT
────────────────────────────────────────────────────────────────────────────────

Streamlit re-runs the entire script on every user interaction. Proper session
state management is critical to maintaining UI continuity and avoiding redundant
LLM calls.

  State keys and their roles:

  api_url          (str)   — The last URL that was analyzed
  has_analyzed     (bool)  — True if the user has run at least one analysis
  analyze_trigger  (bool)  — Set to True by preset buttons / bookmarks;
                              consumed and reset to False in the same run
  parsed_data      (dict)  — The last APISpec returned by parse_api_docs()
                              None when cleared or after a new URL is entered
  request_log      (list)  — List of sandbox request log entries (max 10)
  bookmarks        (list)  — List of {name, url} bookmark objects
  notes            (dict)  — Reserved for future per-API note-taking feature

  SDK and Sandbox state are stored under dynamic keys built from the URL,
  language, style, and endpoint — ensuring each unique combination has its
  own cached state without collisions:

  sdk_{url_key}_{lang}_{style}_{endpoint}  — cached generated code string
  sb_response_{url_key}                    — last Sandbox response dict

  This pattern allows the user to switch between tabs freely without losing
  generated code or the last HTTP response.


────────────────────────────────────────────────────────────────────────────────
  14.6  CACHING STRATEGY
────────────────────────────────────────────────────────────────────────────────

DevFlow uses @st.cache_data with a 10-minute TTL for all LLM calls:

  @st.cache_data(show_spinner=False, ttl=600)
  def _cached_parse(url, use_case):         → calls parse_api_docs()
  def _cached_sdk(name, base, headers, lang, style, ep): → calls generate_sdk()

  Why 10 minutes:
    • Long enough that any user who re-visits a tab or re-renders the page
      does not pay for another LLM call
    • Short enough that documentation changes (rare but possible) are picked up
      in a reasonable timeframe
    • Groq API calls are free-tier limited — caching prevents quota exhaustion

  Cache key design:
    • _cached_parse is keyed on (url, use_case) — same URL with different use
      cases generates separate cached entries with targeted endpoint focus
    • _cached_sdk is keyed on all 6 parameters — a language/style change always
      triggers fresh generation, but the same combination returns instantly

────────────────────────────────────────────────────────────────────────────────
  14.8  ERROR HANDLING & RESILIENCE
────────────────────────────────────────────────────────────────────────────────

DevFlow is designed to degrade gracefully at every failure point:

  Groq API not configured:
    → Sidebar shows "🔴 Groq API key missing" error chip
    → Analyze button blocked with an error message explaining the fix
    → App remains fully navigable; landing page still loads

  Documentation URL unreachable or times out:
    → scraper.fetch_doc_text() raises an exception caught in parse_api_docs()
    → RuntimeError propagated to display_results(), shown to user with st.error()
    → Session state is not polluted — user can try a different URL

  LLM returns malformed JSON:
    → _strip_json_fences() removes any accidental markdown wrapping
    → json.loads() failure caught, empty dict returned
    → _validate_spec() fills in safe defaults — app renders a usable fallback

  Page scraping returns empty content:
    → Fallback APISpec with generic placeholder endpoints is displayed
    → Clear error message: "The page may require authentication or heavy JS"
    → Tips provided: "Try a direct, plain-text REST API reference page"

  Sandbox request fails:
    → Timeout (15s): "Request timed out" shown in error card
    → Connection error: full error message displayed
    → Invalid JSON in headers/params/body: per-field validation with
      specific error messages before the request is ever sent
    → HTML response from API: warning banner explaining the URL is wrong

  LLM primary model fails:
    → groq_client.chat_completion() auto-retries with fallback model
      (llama3-8b-8192) on any exception from the primary (llama-3.3-70b)
    → If fallback also fails: RuntimeError propagated to UI with st.error()

  SDK generation fails:
    → Returns a comment string: "# SDK generation failed. Check Groq API key."
    → Never crashes the tab; user sees the comment in the code block


────────────────────────────────────────────────────────────────────────────────
  14.9  THE GROQ / LLAMA LLM INTEGRATION
────────────────────────────────────────────────────────────────────────────────

All LLM calls in DevFlow route through a single function: chat_completion()
in utils/groq_client.py. This is enforced by design — no LLM call exists
anywhere else in the codebase.

  Primary model:  llama-3.3-70b-versatile
    • 128,000 token context window — can handle very long API documentation
    • 70 billion parameter model — excellent instruction following, JSON output
    • Groq's hardware (LPU) delivers sub-2-second first-token latency

  Fallback model: llama3-8b-8192
    • 8,192 token context window — smaller but faster
    • Used automatically if the primary model call raises any exception
    • Handles simpler documentation parsing tasks adequately

  JSON mode:
    • Enabled for documentation parsing (json_mode=True) — guarantees that
      the response is always valid JSON, even if the model would otherwise
      include prose
    • Disabled for SDK generation (json_mode=False) — code output is not JSON,
      and JSON mode would corrupt it

  Temperature settings:
    • 0.10 for parsing — as deterministic as possible; facts, not creativity
    • 0.15 for SDK generation — slight variance is acceptable for code style
      without sacrificing correctness

  API key management:
    • GROQ_API_KEY loaded from .env via python-dotenv at module import time
    • get_client() returns None if key is missing (never raises at import)
    • All UI paths check get_client() before calling chat_completion()

  14.11  SECURITY CONSIDERATIONS
────────────────────────────────────────────────────────────────────────────────

  ✅ API keys are stored in .env (not hardcoded anywhere in source)
  ✅ .env is never committed to version control (added to .gitignore)
  ✅ Groq key is loaded at runtime only, not exposed in any UI element
  ✅ Sandbox requests are made from the server, not the browser
     → User's API keys are never sent to the client browser
  ✅ No user data is stored on disk — all state is in-memory (session_state)
  ✅ BeautifulSoup parses HTML safely — no eval() or exec() of scraped content
  ✅ No database, no file uploads, no user authentication required

  ⚠ Sandbox sends arbitrary HTTP requests to any URL the user specifies.
    In a multi-tenant deployment, this should be rate-limited and the list of
    allowed hosts should be restricted to prevent SSRF (Server-Side Request Forgery).

  ⚠ GEMINI_API_KEY is present in backend/.env but unused in Python code.
    Remove it from .env to reduce the attack surface if Gemini is not needed.

  ⚠ For deployment: replace .env with platform-level secrets management
    (Streamlit Cloud Secrets, AWS SSM, GCP Secret Manager, etc.)


────────────────────────────────────────────────────────────────────────────────
  14.12  PERFORMANCE CHARACTERISTICS
────────────────────────────────────────────────────────────────────────────────

  Operation                        Typical Duration    Notes
  ───────────────────────────────  ──────────────────  ─────────────────────────
  Documentation fetch (scraper)    0.5 – 2.0 seconds   Depends on doc site speed
  LLM parsing (Groq / LLaMA 3.3)  2.0 – 6.0 seconds   128K context, JSON mode
  SDK generation (Groq / LLaMA)   1.5 – 4.0 seconds   max 2048 tokens output
  Sandbox HTTP request             0.1 – 3.0 seconds   Depends on target API
  Cached results (any above)       < 50 milliseconds   Streamlit @st.cache_data

  The most expensive operation is always the first LLM parsing call for a new
  URL. Subsequent interactions (tab switching, SDK generation for new languages,
  sandbox testing) feel immediate because:
    1. APISpec is cached in session_state after first parse
    2. SDK results are cached by (url, lang, style, endpoint) key
    3. Groq's LPU hardware provides extremely fast inference compared to GPU

  The progress indicator (st.status) during analysis provides user feedback
  with three sequential messages and small delays to feel responsive even
  during the LLM wait.

  14.13  CODE QUALITY & CONVENTIONS
────────────────────────────────────────────────────────────────────────────────

  Python version    : 3.10+  (uses TypedDict, match/case compatible)
  Linting           : Standard Python conventions
  Type hints        : Used throughout utils/ for all function signatures
  Docstrings        : All public functions have docstrings in Google style
  Naming convention : snake_case for functions/variables, PascalCase for types
  Private helpers   : Prefixed with underscore (_build_prompt, _clean_code, etc.)
  Module boundary   : All utils/ functions are exported via utils/__init__.py
                      → app.py imports only from utils, never from submodules directly

  Design patterns used:
    • Singleton pattern: Groq client is initialised once (get_client())
    • TypedDict: APISpec and Endpoint enforce structured data contracts
    • Thin controller: app.py delegates all work to utils/
    • Decorator-based caching: @st.cache_data wraps expensive I/O functions
    • Defensive programming: all external calls wrapped in try/except
